Secret Injection for AI Agent Sandboxes

Research report — How do platforms prevent agents from seeing API keys?

Status — base contract shipped, proxy design proposed

Implemented today: the vault contract — Vault, SecretRef, SecretEnv, ResolvedSecret, SecretRedactor, plus the LocalEnvVault and StaticVault backends.

Not built yet: the proxy-based credential injection, placeholder substitution, and HTTPS-intercepting credential proxy described below. Those are a proposed design; today secrets resolve to a ResolvedSecret value injected directly into the environment.

The Problem

An AI agent running in production needs to call external APIs — Stripe, AWS, databases, etc. The agent's code needs to authenticate. But the agent is an untrusted process: it can run arbitrary code, call printenv, read /proc/self/environ, or log environment variables.

The question: how do you give the agent's code access to credentials without giving the agent itself the ability to read, log, or exfiltrate them?

Host Sandbox (microVM) ┌───────────────────────────────┐ ┌────────────────────────────┐ │ │ │ │ │ Agent (LLM) │ │ python script │ │ "call Stripe to get revenue" │ │ import stripe │ │ │ │ stripe.api_key = ??? │ │ ❌ Agent runs: printenv │ │ │ │ → sees STRIPE_KEY │ │ ❌ Agent runs: printenv │ │ → can exfiltrate it │ │ → sees STRIPE_KEY │ │ │ │ → can exfiltrate it │ └───────────────────────────────┘ └────────────────────────────┘ The traditional approach: inject as env var → agent can read it.

Two Approaches Exist in the Industry

Every platform we researched falls into one of two categories.

Approach 1: Environment Variable Injection (Traditional)

The secret is stored securely (encrypted at rest, IAM-scoped access, rotation). At runtime, it's decrypted and injected as a plain environment variable. Once the process starts, any code can read it.

This does not prevent agent exfiltration. printenv, os.environ, cat /proc/self/environ all expose the raw secret. These platforms solve storage and delivery — not in-process extraction.

Approach 2: TLS Credential Proxy (New, Agent-Specific)

The real secret never enters the sandbox. The sandbox receives a meaningless placeholder. A host-side proxy intercepts outbound HTTPS traffic, matches against approved destinations, and swaps the placeholder for the real credential in the request headers before forwarding.

This actually prevents exfiltration. printenv shows msb_placeholder_a7f3... or proxy-managed. There is nothing useful to extract. The real secret exists only in the host-side proxy process.

Platform-by-Platform Analysis

Microsandbox

Approach: TLS MITM proxy with placeholder substitution

When you configure a secret with --secret STRIPE_KEY=sk_live_...@api.stripe.com, microsandbox generates a random placeholder like msb_placeholder_a7f3... and injects that into the VM as the environment variable. The real secret never enters the guest. A transparent MITM proxy intercepts outbound HTTPS, and when the destination matches an allowed host, swaps the placeholder for the real credential in transit. DNS rebinding protection and cloud metadata blocking activate automatically.

  • printenv → shows msb_placeholder_a7f3...
  • /proc/self/environ → shows the placeholder
  • Even with full code execution, nothing useful to exfiltrate

Limitation: TLS interception breaks certificate pinning. If the target API pins certs, requests fail.

Docs · GitHub

Docker Sandboxes (sbx)

Approach: HTTPS proxy + sentinel values + OS keychain

Credentials are stored in the host OS keychain via sbx secret set. Inside the VM, the environment variable holds only a sentinel value like "proxy-managed". All outbound traffic routes through a host-side proxy at host.docker.internal:3128. The proxy terminates TLS, inspects the request, looks up the matching credential from the keychain, overwrites the auth header, then re-encrypts and forwards upstream. A per-session CA certificate is installed inside the VM.

  • printenv → shows proxy-managed
  • Real credential lives only in the host OS keychain
  • Sandbox has isolated network — cannot reach host localhost

Docs · Blog

Cloudflare Sandboxes

Approach: Outbound Workers + ephemeral CA

Uses "Outbound Workers" that run outside the sandbox. Credentials are injected into requests by the Worker — the sandbox never receives them. Supports dynamic per-instance egress policies, allow/deny lists, and runtime policy changes without restart. Same architectural pattern as microsandbox and Docker sbx.

Blog · Changelog

Vercel Sandbox

Approach: TLS MITM proxy + ephemeral ECDSA CA

Per-sandbox ephemeral ECDSA CA (P-256) with 24-hour TTL. Supports path/method/header matching rules for selective credential injection. The proxy injects credentials into HTTP headers in transit. The sandbox never holds the real credential.

Docs · Changelog

Infisical Agent Vault

Approach: Open-source HTTPS_PROXY + TLS interception

Standalone Go proxy that can be deployed alongside any sandbox (E2B, Daytona, custom). Sets HTTPS_PROXY and injects CA certs for Python, Node, curl, Go, Deno. AES-256-GCM encryption at rest. Scoped sessions with audit logging. Retrofits credential brokering onto platforms that lack it natively.

Blog · GitHub

Modal

Approach: Plain environment variables

modal.Secret objects are key-value pairs injected as standard env vars. Modal's own documentation explicitly warns: "Never put secrets inside a sandbox. API keys, tokens, database credentials, and other secrets injected into a sandbox can be read and exfiltrated." They recommend blocking network access and using human-in-the-loop approval as mitigations.

Docs — Secrets · Docs — Sandboxes

Daytona

Approach: Plain environment variables

Workspaces accept environment variables via SDK or API. These are set as standard env vars inside the sandbox container. No credential proxy or TLS interception. Can be combined with Infisical Agent Vault as a workaround, but nothing built-in.

Docs

Fly.io / Fly Machines

Approach: Encrypted vault → env vars at boot

fly secrets set stores values in an encrypted vault. At boot, the Fly agent decrypts and injects as env vars. The API servers can only encrypt, never decrypt. Secret values are never logged. But once the Machine is running, secrets are standard env vars. Fly docs explicitly state: "People with deploy access can deploy code that reads secret values and prints them to logs."

Docs

DigitalOcean App Platform

Approach: Encrypted env vars

"Encrypted" variables are encrypted at rest and hidden from UI/API/logs. At runtime, the decrypted value is a plain environment variable. The "encrypted" flag protects the value in the control plane only — not from the running process. Uses Kata Containers for workload isolation.

Docs

AWS ECS / Lambda

Approach: Secrets Manager → env vars at task/invocation start

ECS: Task definition references a Secrets Manager ARN. ECS agent fetches the value and injects as an env var. The execution role needs secretsmanager:GetSecretValue.

Lambda: Env vars encrypted at rest with KMS, decrypted at invocation. AWS docs acknowledge: "anyone who has access to the Lambda function code can extract all the secrets with a simple print or console.log command."

ECS Docs · Lambda Docs

HashiCorp Vault

Approach: Sidecar → plaintext files on shared volume

The Vault Agent Injector (Kubernetes) injects an init container that authenticates, fetches secrets, and writes them to /vault/secrets/ on a shared memory-backed volume. The application reads secrets from these files. Any process in the pod can read them. Vault provides secure delivery, rotation, and audit logging — but once the secret is on disk, it's accessible.

How the TLS Proxy Works (Technical Detail)

1. Secret configured: msb create python --secret STRIPE_KEY=sk_live_abc123@api.stripe.com 2. Sandbox boots: - VM env var: STRIPE_KEY=msb_placeholder_a7f3e9... - MITM CA cert installed in VM trust store - DNS rebinding protection enabled - Cloud metadata endpoint (169.254.169.254) blocked 3. Agent's code runs: stripe.api_key = os.environ["STRIPE_KEY"] # → "msb_placeholder_a7f3e9..." stripe.charges.list() # → HTTPS request to api.stripe.com with placeholder in auth header 4. Host-side proxy intercepts: - Terminates TLS (using MITM CA) - Sees: Authorization: Bearer msb_placeholder_a7f3e9... - Destination: api.stripe.com ← matches allowed host - Swaps: Authorization: Bearer sk_live_abc123 - Re-encrypts with real api.stripe.com cert - Forwards to Stripe 5. Response flows back through proxy to sandbox normally. 6. If agent runs printenv: STRIPE_KEY=msb_placeholder_a7f3e9... ← useless 7. If agent tries to send placeholder to evil.com: - Proxy sees destination: evil.com ← not in allowed hosts - Does NOT swap placeholder → evil.com receives meaningless string

Limitations of the Proxy Approach

  • Breaks TLS certificate pinning — the MITM CA is not the real CA. APIs that pin certificates will reject the connection. Most public APIs (Stripe, AWS, etc.) don't pin, but some financial/enterprise APIs do.
  • HTTP/HTTPS only — the proxy intercepts HTTP headers. Binary protocols, mutual TLS client certificates, SSH keys, or database connection strings need different handling (e.g., separate proxy for Postgres wire protocol).
  • Performance overhead — every HTTPS request goes through an extra TLS termination and re-encryption. Negligible for API calls, noticeable for high-throughput data pipelines.
  • CA trust management — the sandbox must trust the proxy's CA. Different languages/runtimes have different trust stores (Python certifi, Node NODE_EXTRA_CA_CERTS, Go system store, Java keystore). Infisical Agent Vault handles this for Python, Node, curl, Go, and Deno automatically.

Emerging Standard: IETF CB4A

An Internet-Draft (draft-hartman-credential-broker-4-agents-00, published 29 March 2026) proposes a standardized Credential Broker for Agents (CB4A) architecture. As of June 2026 it remains at revision -00 — an individual submission by Kenneth G. Hartman (SANS), not yet adopted by an IETF working group and with no formal standing in the IETF standards process. The draft is active but set to expire 30 September 2026; its status should be re-verified before citing. It builds on:

  • SPIFFE/SPIRE for workload identity
  • DPoP (RFC 9449) for sender-constrained tokens
  • NIST SP 800-207 for policy separation

Defines three proxy models and a tiered approval framework. If ratified, this would standardize the proxy-based approach that microsandbox, Docker sbx, Cloudflare, and Vercel have independently converged on.

Implications for Cayu

Cayu's Vault primitive should support the proxy-based approach as the default for sandbox runners. The design:

  1. Tools and environments declare secret requirements via SecretRef — name + allowed hosts
  2. The runtime resolves SecretRef through the Vault → gets the real value
  3. The runtime passes a placeholder into the sandbox as an env var
  4. A credential proxy (microsandbox's built-in, or Infisical Agent Vault) intercepts outbound HTTPS
  5. Proxy swaps placeholder for real value on approved hosts only
  6. Agent sees STRIPE_KEY=msb_placeholder_... — useless if exfiltrated

Proposed per-runner mapping (none of this placeholder/proxy wiring exists in the cayu runners yet): MicrosandboxRunner could build on microsandbox's native --secret proxy. DockerRunner has no native equivalent — it injects plain env via docker run -e today (copy_runner_env(env, inherit_env=False)), so it would need an external credential proxy/sidecar. For LocalRunner (development), plain env vars are fine — the developer is trusted.

Runner status, to be precise. MicrosandboxRunner and DockerRunner are implemented in the current codebase (src/cayu/runners/microsandbox.py, src/cayu/runners/docker.py). What is not yet built is the proxy-based secret injection inside these runners — the placeholder substitution and HTTPS-intercepting credential proxy. Today there is no runtime secret resolution on the exec path at all: ExecCommandTool forwards the plain env it is given, and resolving a SecretRef through the Vault is the application's job via Environment.resolve_secret — the runner just receives the resulting plain env vars.
The framework advantage. Because Cayu owns the Environment → Vault → Runner pipeline, it can transparently upgrade from plain env vars (LocalRunner, dev) to proxy-based injection (MicrosandboxRunner, production) without changing agent code or tool declarations. The agent never knows how secrets are delivered — it just declares what it needs.
